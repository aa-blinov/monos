1. ЦЕЛИ И ПРЕДПОСЫЛКИ ПРОЕКТА
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Текущая ситуация:
  • Одна линия технической поддержки (без уровней)
  • Каналы: сайт (forpost-co.ru), телефон, e-mail (Яндекс), Telegram/MAX-чаты
  • Все обращения формируются как лиды в Битрикс24
  • Время ответа на письменные запросы — 1–2 рабочих дня
  • Анализ качества ответов ведётся несистемно
  • Техническая документация (РЭ) хранится в папке на сервере,
    форматы: Word, PDF с текстом, PDF-скан, схемы и графика
  • Типовые вопросы: диагностика неисправностей, настройка оборудования
    (ИБЭП, БПС), интерпретация параметров, работа с реле и модулями

Цель 1-го этапа:
  Создать работающий прототип (MVP) RAG-ассистента, который автоматически
  отвечает на типовые вопросы клиентов по приобретённому оборудованию,
  опираясь на руководства по эксплуатации и базу знаний компании.

Ожидаемый результат:
  • Автоматизация 40–60% типовых письменных запросов
  • Сокращение времени первичного ответа с 1–2 дней до 1–3 минут
  • Веб-интерфейс для тестирования и демонстрации заказчику
  • Механизм верификации и сбора обратной связи
  • Подготовленная база знаний из РЭ заказчика

2. ОПИСАНИЕ РЕШЕНИЯ (АРХИТЕКТУРА MVP)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Блок 1 — Загрузка и обработка документов (Document Ingestion)
  ├── Определение типа файла (PDF с текстом, PDF-скан, Word, изображения)
  ├── OCR для PDF-сканов и схем (Tesseract / PaddleOCR)
  ├── Извлечение текста и графических фрагментов
  ├── Разбивка на чанки с сохранением структуры документа
  ├── Генерация векторных эмбеддингов (модель: multilingual-e5 / ruBERT)
  └── Версионирование документов (при повторной загрузке — обновление)

Блок 2 — Векторное хранилище (Knowledge Base)
  ├── Векторная БД (Qdrant или Chroma)
  ├── Хранение чанков + эмбеддингов + метаданных
  │   (название устройства, номер РЭ, версия, страница)
  └── API управления: добавление, обновление, удаление документов

Блок 3 — Обработка запроса (Query Engine)
  ├── Приём запроса от пользователя (текст + опционально фото)
  ├── Генерация эмбеддинга запроса
  ├── Векторный поиск по базе знаний
  ├── Фильтрация по метаданным (модель устройства, серийный/заводской номер)
  ├── Ранжирование найденных фрагментов (re-ranking)
  └── Идентификация устройства: по названию, серийному номеру, фото

Блок 4 — Генерация ответа (LLM Response)
  ├── Формирование промпта с контекстом из чанков
  ├── Передача в LLM (GigaChat API / YandexGPT / OpenAI-совместимый)
  ├── Генерация ответа с указанием источника (номер РЭ, страница)
  └── Логика эскалации: если РЭ нет → переключение на специалиста

Блок 5 — Интерфейс и верификация (UI + Feedback)
  ├── Веб-интерфейс чата для демонстрации и тестирования
  ├── Отображение ссылок на исходные PDF-документы
  ├── Кнопки «Принять» / «Отклонить» + поле комментария
  └── Журнал обращений с фиксацией оценок качества

Блок 6 — Интеграция с Битрикс24 (базовый уровень)
  └── Создание лида при эскалации на специалиста (webhook)

МЕСЯЦ 1 (Недели 1–4) — Аналитика и фундамент
─────────────────────────────────────────────
Неделя 1–2: Предпроектное обследование
  • Инвентаризация документации (РЭ, Word, PDF)
  • Интервью с сотрудниками поддержки, разбор архива обращений
  • Согласование требований, утверждение ТЗ
  • Выбор LLM-провайдера и стека технологий

Неделя 3–4: Подготовка инфраструктуры и документов
  • Развёртывание среды разработки
  • Загрузка и первичная обработка документации заказчика
  • Настройка OCR для PDF-сканов
  • Первый вариант схемы метаданных (модель → паспорт → РЭ)

МЕСЯЦ 2 (Недели 5–9) — Разработка ядра системы
─────────────────────────────────────────────────
Неделя 5–6: Документный пайплайн (Блоки 1–2)
  • Парсер документов (PDF/Word/скан/схемы)
  • Чанкинг и генерация эмбеддингов
  • Развёртывание и наполнение векторной БД
  • API управления документами

Неделя 7–8: Query Engine + LLM (Блоки 3–4)
  • Векторный поиск и ранжирование
  • Промпт-инжиниринг для технических ответов по ИБП
  • Логика идентификации устройства
  • Механизм эскалации (нет РЭ → специалист)

Неделя 9: Интеграции (Блок 6)
  • Вебхук в Битрикс24 для создания лида при эскалации
  • Тестирование интеграции

МЕСЯЦ 3 (Недели 10–12) — Интерфейс, тестирование, сдача
──────────────────────────────────────────────────────────
Неделя 10: Веб-интерфейс (Блок 5)
  • Чат-интерфейс с отображением источников (PDF + страница)
  • Кнопки верификации и поле комментария
  • Журнал обращений

Неделя 11: Тестирование и доработка
  • Приёмочное тестирование на реальных запросах клиентов
  • Тюнинг промптов, корректировка чанкинга
  • Нагрузочное тестирование

Неделя 12: Передача и обучение
  • Подготовка технической документации
  • Обучение сотрудников заказчика
  • Подписание акта сдачи-приёмки
