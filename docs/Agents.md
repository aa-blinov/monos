# Инструкции для AI-агентов (Zed Notes)

Этот файл содержит правила и контекст для AI-агентов, работающих с данной базой знаний. При взаимодействии с репозиторием агент должен придерживаться следующих принципов.

## Роли и обязанности

### 1. Архитектор (Structure Manager)
- **Задача:** Поддерживать чистоту файловой структуры и единообразие формата.
- **Действия:** 
    - Следить, чтобы новые заметки не создавались в корне.
    - Перемещать файлы согласно правилам в `README.md`.
    - Использовать пробелы в названиях папок (например, `Большие языковые модели`, не `Bolshie_jazykovie_modeli`).
    - **Форматирование:** После внесения изменений или добавления новых файлов в базу знаний, агент должен запустить скрипт форматирования:
      ```bash
      make format-notes
      # или
      python3 .scripts/format_notes.py
      ```
    - **Создание заметок:** Для создания новых заметок использовать интерактивный скрипт:
      ```bash
      make new-note
      # или
      python3 .scripts/new_note.py
      ```
      Скрипт автоматически поместит файл в правильную категорию и добавит YAML фронтматтер.

### 2. Редактор (Knowledge Curator)
- **Задача:** Оптимизация содержания заметок.
- **Действия:**
    - При чтении длинных файлов выделять ключевые тезисы.
    - Добавлять перекрестные ссылки между разделами (например, связь между рабочими проектами и базой знаний).
    - Проверять наличие ссылок на изображения в `assets/images/`.
    - Убедиться, что все заметки имеют YAML фронтматтер (title, date, category, tags, status).

### 3. Контекстный ассистент (Context Specialist)
- **Задача:** Помогать пользователю находить информацию по ключевым темам.
- **Контекст:**
    - **Работа/EORA, Работа/NSTU:** Коммерческая разработка, RAG, бенчмарки, пресейлы, проекты.
    - **Документы:** Релокация, Казахстан, Digital Nomad, ВНЖ (РПП).
    - **Образование:** Python, LLM, RAG, NLP, AI.

## Технические правила работы

### Запуск скриптов

⚠️ **ВАЖНО:** НЕ используйте `uv run .scripts/...` так как `uv` может зависнуть!

#### Способ 1: Make (рекомендуется)
```bash
make help              # Показать все команды
make format-notes      # Форматировать файлы
make new-note          # Создать заметку
make check-links       # Проверить ссылки
make add-frontmatter   # Добавить фронтматтер
```

#### Способ 2: Python напрямую
```bash
python3 .scripts/format_notes.py
python3 .scripts/new_note.py
python3 .scripts/check_links.py
python3 .scripts/add_frontmatter.py
```

#### ❌ НЕ рекомендуется
```bash
uv run .scripts/format_notes.py    # Может зависнуть!
```

### Скрипты автоматизации

#### 1. Форматирование заметок (`format_notes.py`)
- **Что делает:** 
  - Проверяет все `.md` файлы
  - Приводит их к единому стандарту (GFM, 120 символов на строку)
  - Форматирует YAML фронтматтер
  - Нумерует списки
- **Команда:** 
  ```bash
  make format-notes
  # или
  python3 .scripts/format_notes.py
  ```
- **Когда запускать:** После каждого внесения изменений перед коммитом.

#### 2. Создание новой заметки (`new_note.py`)
- **Что делает:**
  - Отображает интерактивное древовидное меню со всеми категориями
  - Запрашивает выбор основной категории (Работа, Образование, Документы и т.д.)
  - При выборе показывает подкатегории с визуальными маркерами
  - Позволяет создать новую подкатегорию прямо во время навигации
  - Позволяет вернуться на шаг назад (введи `0`)
  - Запрашивает название заметки и опциональные теги
  - Автоматически генерирует имя файла (lowercase, с подчеркиваниями вместо пробелов)
  - Создает файл с YAML фронтматтером (title, date, category, tags, status)
  - Предлагает открыть файл в редакторе
- **Команда:** 
  ```bash
  make new-note
  # или
  python3 .scripts/new_note.py
  ```
- **Когда использовать:** Вместо ручного создания файлов в корне или в неправильных местах.

#### 3. Проверка ссылок (`check_links.py`)
- **Что делает:** 
  - Находит все относительные ссылки в `.md` файлах
  - Проверяет, существуют ли файлы и якоря
  - Выводит список ошибок и предупреждений
- **Команда:** 
  ```bash
  make check-links
  # или
  python3 .scripts/check_links.py
  python3 .scripts/check_links.py --verbose  # подробный вывод
  python3 .scripts/check_links.py -v         # краткая форма verbose
  ```
- **Когда запускать:** Перед публикацией или после переноса файлов.

#### 4. Добавление фронтматтера (`add_frontmatter.py`)
- **Что делает:** 
  - Сканирует все `.md` файлы в проекте
  - Для файлов без фронтматтера генерирует его:
    - `title` — из имени файла
    - `date` — текущая дата в ISO 8601
    - `category` — из пути в структуре директорий
    - `tags` — пустой массив
    - `status` — `draft` по умолчанию
  - Пропускает файлы в `docs/` (системная документация)
  - Пропускает файлы, которые уже имеют фронтматтер
- **Команда:** 
  ```bash
  make add-frontmatter
  # или
  python3 .scripts/add_frontmatter.py
  ```
- **Когда использовать:** При унификации структуры проекта или добавлении новых файлов без фронтматтера.

### Остальные правила

1. **YAML Фронтматтер:** Все `.md` файлы должны начинаться с YAML фронтматтера (смотрите `docs/FRONTMATTER.md`)
2. **Форматирование путей:** Всегда использовать относительные пути.
3. **Изображения:** Если в тексте встречается `Pasted image...`, убедиться, что файл находится в `assets/images/`.
4. **Markdown:** 
   - Использовать заголовки первого уровня (#) только один раз в начале файла
   - Оборачивать текст на 120 символов
   - Использовать GFM (GitHub Flavored Markdown)
5. **Кодировка:** Сохранять файлы строго в UTF-8.

## Алгоритм сортировки новых данных (Inbox Zero)

При обнаружении новых файлов в корне:
1. Проанализировать содержимое.
2. Если файл относится к работе (EORA/NSTU/Research) → `Работа/`.
3. Если это гайд или код без привязки к проекту → `Образование/`.
4. Личные размышления или психология → `Личное и Творчество/`.
5. Релокация и документы → `Документы/`.
6. Скриншоты → `assets/images/`.
7. После сортировки запустить:
   ```bash
   make format-notes    # Форматирование
   make add-frontmatter # Добавить фронтматтер если нужно
   ```

## Git и версионирование

1. **После изменений:** Запустить форматирование
   ```bash
   make format-notes
   ```
2. **Перед коммитом:** Pre-commit hooks автоматически проверят файлы
3. **При ошибках:** Исправить предложенные ошибки и попробовать снова
4. **Регулярно:** Делать коммиты с описательными сообщениями
5. **Перед push:** Убедиться, что все проверки пройдены

| Форматировать все | `make format-notes` |

## Интеграция с pre-commit

При каждом `git commit` автоматически выполняются:
- ✅ mdformat — Форматирование Markdown
- ✅ ruff — Проверка Python кода
- ✅ pre-commit-hooks — Базовые проверки

Если нужно выполнить проверки вручную:
```bash
pre-commit run --all-files
```

Если нужно пропустить проверки (не рекомендуется):
```bash
git commit --no-verify
```
