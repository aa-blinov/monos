## Интервью-кейс: "AI-ассистент для финансовых транзакций"

**Цель задания:** Оценить способность кандидата спроектировать ядро AI-продукта, используя стек вакансии (Python, FastAPI, OpenAI, LangChain, Celery). Проверить не только навыки работы с LLM, но и "production-ready" мышление (безопасность, асинхронность, тестирование).

**Контекст (Легенда для кандидата):** "Мы делаем 'сердце' нашего продукта XINDATA. Представьте, что у нас есть база данных с финансовыми транзакциями *всех* наших пользователей. Мы хотим создать AI-сервис, который будет отвечать на вопросы конкретного пользователя о *его* финансах на естественном языке".

## Блок 1: Схема данных и основная задача

(Показываем кандидату эту простую схему или описываем ее)

**Дано:**

1. **SQL База данных** (например, PostgreSQL) с *одной* большой таблицей `transactions`:

   - `user_id` (int) — ID пользователя.
   - `datetime` (timestamp) — Дата и время операции.
   - `sum` (decimal) — Сумма операции.
   - `direction` (string) — Направление ('income' или 'expense').
   - `purpose` (string) — Назначение платежа (e.g., "Аренда офиса", "Зарплата").
   - `contragent` (string) — Контрагент (e.g., "ООО 'Ромашка-Трейд'", "ИП Иванов").

1. **Входные данные сервиса:**

   - `user_id` (e.g., 123)
   - `question` (e.g., "Сколько я потратил на Ромашку в прошлом месяце?")

1. **Ожидаемый выход:**

   - `Человеко-читаемый ответ` (e.g., "В прошлом месяце вы потратили 45 000 руб. на контрагентов, похожих на 'Ромашка'.")

**Основной вопрос (Фильтр на AI/LLM):** "Как бы вы спроектировали систему, которая решает эту задачу, используя наш стек (Python, OpenAI, LangChain)? Опишите основные компоненты и как они будут взаимодействовать".

- **Ожидаемый ответ:** Кандидат должен быстро прийти к идее **NL-to-SQL** (Natural Language to SQL).
- **Идеальный ответ:** Упоминание **LangChain SQL Agents** (или `create_sql_agent`), передача схемы БД в промпт, возможное упоминание `SQLDatabaseChain`.

## Блок 2: Углубленные вопросы (Проверка Middle+)

*Если кандидат предложил NL-to-SQL, продолжаем:*

**1. Безопасность и Авторизация (Критически важно):**

- "В этой таблице лежат данные *всех* пользователей. Как вы *гарантируете*, что пользователь `user_id: 123`, задавая вопрос, не сможет увидеть данные пользователя `user_id: 456`? Что, если он прямо спросит: 'Покажи траты пользователя 456'?"
- **Ожидаемый ответ:** Нельзя просто доверять LLM. Нужно *принудительно* добавлять `WHERE user_id = 123` к *каждому* сгенерированному SQL-запросу. Обсуждение, где это делать: в промпте агента, или (что надежнее) через парсинг и валидацию сгенерированного SQL *до* его выполнения.

**2. Сложная логика (Fuzzy Search / RAG):**

- "В базе у нас `ООО 'Ромашка-Трейд'`, а пользователь спрашивает про 'Ромашку'. Простой `LIKE '%Ромашка%'` может быть неэффективным. Как улучшить поиск по контрагентам или назначениям платежа, чтобы он был 'умным'?"
- **Ожидаемый ответ:** Обсуждение **"сложной цепочки" (Chain)**. Например:
  1. Сначала RAG-поиск: Найти наиболее релевантное *точное* имя (`"ООО 'Ромашка-Трейд'"`) по запросу (`"Ромашка"`) в отдельной, векторизованной базе контрагентов.
  1. И только *потом* использовать это точное имя в NL-to-SQL агенте.

**3. Проектирование API (FastAPI):**

- "Отлично, с логикой разобрались. Как 'обернуть' это в API на **FastAPI**? Какие эндпоинты, методы (GET/POST) и Pydantic-модели (для запроса и ответа) вы бы создали?"
- **Ожидаемый ответ:** `POST /ask` с Pydantic-моделью `Query(user_id: int, question: str)`. Ответ — `Answer(text: str)`.

**4. Асинхронность и Production (Celery):**

- "Вызов LLM-агента и выполнение SQL-запроса по большой таблице могут занять 10-15 секунд. HTTP-запрос 'повиснет', Telegram-бот отвалится по таймауту. Как, используя наш стек, решить эту проблему?"
- **Ожидаемый ответ:** Использование **Celery** + (RabbitMQ/Redis).
  1. API на FastAPI *мгновенно* принимает запрос.
  1. Создает задачу для Celery (`process_query.delay(user_id, question)`).
  1. Отвечает пользователю (e.g., `"Принял, думаю…"`).
  1. Celery-воркер выполняет тяжелую задачу и отправляет результат (например, обратным вызовом в Telegram API или через WebSocket).

**5. Тестирование (Must-Have из вакансии):**

- "Как бы вы тестировали эту систему? Особенно самую 'плавающую' часть — LLM-агента. Как убедиться, что после обновления промпта мы не сломали старые запросы?"
- **Ожидаемый ответ:** Юнит-тесты для API и воркеров. Для LLM — создание "Golden Set" (эталонного набора): `[("Вопрос 1", "Ожидаемый SQL 1"), ("Вопрос 2", "Ожидаемый SQL 2")]`. Прогон этого набора в CI/CD для проверки регрессии.

## Блок 3: План "Б" (Если кандидат не догадался до NL-to-SQL)

*Если кандидат не предлагает NL-to-SQL, а пытается парсить вопрос регулярками.*

**Упрощение задачи (Проверка навыков анализа данных):** "Хорошо, давайте сильно упростим. Забудем про SQL. У нас есть **CSV-файл** с транзакциями *только для этого пользователя* (поля: `datetime, sum, direction, purpose, contragent`)."

"Как, используя Python (возможно, `pandas`) и *простой* вызов LLM (OpenAI API, *без* агентов), вы бы ответили на тот же вопрос: 'Сколько я потратил на Ромашку в прошлом месяце?'"

- **Ожидаемый ответ:**
  1. Загрузить CSV в `pandas.DataFrame`.
  1. Отфильтровать DataFrame (по дате, по `direction == 'expense'`).
  1. Возможно, сделать строковый поиск по `contragent`.
  1. Сформировать из этих данных *контекст* (e.g., "Вот данные о тратах: …") и "скормить" его LLM вместе с вопросом, чтобы она дала красивый ответ и посчитала итог.
