# Архитектура мультиагентных систем на базе LLM

В последние годы один из наиболее перспективных и активно развивающихся направлений в области искусственного интеллекта — это создание систем, состоящих из нескольких взаимодействующих агентов, каждый из которых работает на основе больших языковых моделей. Такие мультиагентные системы (MAS, Multi-Agent Systems) способны решать сложные задачи, которые находятся за пределами возможностей единого агента. Эта статья рассмотрит архитектуру, принципы проектирования и лучшие практики построения эффективных мультиагентных систем.

## Введение: от монолитного агента к экосистеме

Первые попытки создания AI-агентов на основе LLM часто следовали простой схеме: один агент, один набор инструментов, одна цель. Такой подход работает для узконаправленных задач, но быстро натыкается на ограничения при масштабировании. Основные проблемы единого агента:

1. **Разделение внимания:** Один агент не может одновременно хорошо разбираться в разных доменах (финансы, код, право, маркетинг)
2. **Масштабируемость токенов:** С добавлением новых инструментов context window быстро переполняется
3. **Надежность:** Ошибка на одном этапе разрушает весь результат
4. **Специализация:** Нет возможности оптимизировать разные части под их специфику

Мультиагентные системы решают эти проблемы через **разделение труда**: каждый агент отвечает за узкую область, агенты общаются между собой, координация происходит на более высоком уровне.

## Часть 1: Ключевые архитектурные паттерны

### Паттерн 1: Координатор и специалисты (Coordinator-Specialist)

Это наиболее распространенный паттерн в современных системах с LLM (Anthropic, 2024; Building Effective Agents). Один координирующий агент распределяет задачи между специализированными агентами.

**Структура:**
```
Пользователь
    ↓
[Координатор]
    ├→ [Агент_финансы]
    ├→ [Агент_код]
    ├→ [Агент_анализ]
    └→ [Агент_письмо]
    ↓
Результат
```

**Как работает:**
1. Пользователь дает задачу координатору
2. Координатор анализирует задачу, определяет требуемые компетенции
3. Координатор распределяет подзадачи между специалистами
4. Каждый специалист работает независимо с собственным набором инструментов
5. Координатор собирает результаты и формирует финальный ответ

**Преимущества:**
- Простота реализации
- Легко масштабировать (добавить нового специалиста = добавить новый агент)
- Хорошо работает для структурированных задач

**Ограничения:**
- Координатор может "развалить" задачу неправильно
- Специалисты не могут напрямую общаться (все идет через координатора)
- Не эффективно для сложного взаимозависимого анализа

**Пример из реальности:** Компания получает резюме кандидата. Координатор направляет его к:
- Агенту HR (проверка квалификации)
- Агенту техническому (оценка навыков)
- Агенту культуре (соответствие ценностям)
Затем собирает итоговое заключение.

### Паттерн 2: Иерархический (Hierarchical)

Агенты организованы в иерархию с несколькими уровнями принятия решений. Такая архитектура отражена в исследованиях по multi-agent planning (Wei et al., 2022; Composable Code Search) и часто используется в больших организационных системах.

**Структура:**
```
[Главный агент]
    ├→ [Менеджер анализа]
    │   ├→ [Аналитик_1]
    │   └→ [Аналитик_2]
    └→ [Менеджер исполнения]
        ├→ [Исполнитель_1]
        └→ [Исполнитель_2]
```

**Как работает:**
1. Главный агент принимает высокоуровневое решение (ЧТО делать)
2. Менеджеры среднего уровня определяют КАК это делать
3. Рабочие агенты на нижнем уровне ВЫПОЛНЯЮТ конкретные действия

**Преимущества:**
- Разделение уровней абстракции
- Хорошо отражает реальные организационные структуры
- Упрощает масштабирование для больших систем

**Ограничения:**
- Более сложная в реализации
- Возможны нарушения в цепи командования
- Требует четкого определения интерфейсов между уровнями

**Пример:** Система для управления кампанией:
- Главный агент решает: "Провести летнюю маркетинговую кампанию"
- Менеджер маркетинга определяет: "Нужны соцсети и email"
- Email агент пишет письма, соцсети агент создает посты

### Паттерн 3: Сетевой (Network/Mesh)

Агенты расположены как в сети без центральной координации. Каждый агент может общаться с любым другим. Этот подход исследуется в работах по distributed consensus algorithms (Ren & Beard, 2008) и находит применение в peer-to-peer системах.

**Структура:**
```
[Агент_1] ←→ [Агент_2]
   ↓      ✕      ↓
[Агент_3] ←→ [Агент_4]
```

**Как работает:**
1. Агенты инициируют общение по мере необходимости
2. Нет центрального координатора
3. Решения принимаются через консенсус или переговоры
4. Система саморегулируется

**Преимущества:**
- Максимальная гибкость
- Отказоустойчивость (выход одного агента не разрушает систему)
- Хорошо для сложных, слабо структурированных задач

**Ограничения:**
- Сложно отладить
- Может быть неопределенность в выборе пути
- Требует механизмов предотвращения циклических зависимостей

**Пример:** Система для научного исследования:
- Агент математики и агент физики обсуждают формулировку задачи
- Агент экспериментов и агент анализа вместе проектируют эксперимент
- Все четыре агента вместе интерпретируют результаты

### Паттерн 4: Конвейерный (Pipeline)

Агенты расположены в последовательности, каждый обрабатывает выход предыдущего. Классический паттерн в системах обработки данных и LLM приложениях.

**Структура:**
```
Ввод → [Агент_1] → [Агент_2] → [Агент_3] → [Агент_4] → Вывод
```

**Как работает:**
1. Первый агент получает исходные данные
2. Обрабатывает их и передает следующему
3. Каждый последующий агент получает входные данные от предыдущего
4. Последний агент выдает финальный результат

**Преимущества:**
- Простота понимания и отладки
- Естественно отражает процессы обработки данных
- Легко добавлять новые этапы

**Ограничения:**
- Жесткая последовательность
- Ошибка на одном этапе влияет на все последующие
- Не подходит для параллельной работы

**Пример:** Система для обработки документов:
1. Агент OCR - преобразует изображение в текст
2. Агент парсинга - выделяет структуру документа
3. Агент классификации - определяет тип документа
4. Агент извлечения - вычленяет ключевую информацию

## Часть 2: Механизмы коммуникации

### Синхронная коммуникация (Request-Response)

Агент А отправляет запрос агенту Б и ждет ответа прежде, чем продолжить работу.

**Применение:**
- Когда результат одного агента критичен для другого
- Для управления временем выполнения
- Для гарантии получения ответа

**Недостатки:**
- Может привести к блокировкам
- Если один агент медленный, замораживается весь процесс
- Возможны циклические зависимости (deadlock)

### Асинхронная коммуникация (Event-based)

Агент А отправляет событие и продолжает работу. Агент Б обрабатывает событие, когда ему удобно.

**Применение:**
- Для параллельной работы
- Когда результат может быть обработан позже
- Для высоконагруженных систем

**Преимущества:**
- Лучше масштабируется
- Более устойчива к отказам
- Позволяет обрабатывать неожиданные события

### Очередь сообщений

Агенты отправляют сообщения в общую очередь, из которой их берут другие агенты.

**Применение:**
- Для больших систем с многими агентами
- Когда порядок обработки важен
- Для дублирования и отказоустойчивости

### Claude-specific: Реализация синхронной и асинхронной коммуникации

**Синхронная коммуникация с Claude (использование tool_use):**
```python
# Синхронный вызов агента-специалиста
def coordinator_sync(user_task):
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=2048,
        system="You are a coordinator. Decompose tasks and call specialists.",
        tools=[
            {
                "name": "call_financial_specialist",
                "description": "Call financial specialist agent",
                "input_schema": {"type": "object", "properties": {...}}
            }
        ],
        messages=[{"role": "user", "content": user_task}]
    )
    
    # Обработайте tool_use блоки
    for block in response.content:
        if block.type == "tool_use":
            specialist_result = call_tool(block.name, block.input)
            # Передайте результат обратно в Claude для синтеза
```

**Асинхронная коммуникация (через очереди):**
```python
# Агент отправляет сообщение в очередь
queue.put({
    "agent_id": "financial_specialist",
    "task": "Analyze Q2 revenue",
    "priority": "high",
    "timeout": 30
})

# Другой агент обрабатывает, когда готов
def worker():
    while True:
        message = queue.get()
        response = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            messages=[{"role": "user", "content": message["task"]}]
        )
        message["result"] = response.content
        result_queue.put(message)
```

**Рекомендация:** Для MAS с Claude используйте tool_use для критичных синхронных операций и асинхронные очереди для фоновых задач, это снижает стоимость при сохранении надежности.

## Часть 3: Управление состоянием и контекстом

### Проблема контекстного окна

По мере работы системы контекст каждого агента растет:
- История разговора увеличивается
- Результаты других агентов добавляются в контекст
- Каждое новое взаимодействие добавляет токены

При контексте 4K это становится критичным быстро. Даже с 128K контекстом (GPT-4 Turbo) при активной работе 10 агентов история может заполнить все место.

### Решение 1: Резюмирование контекста

Периодически берется история, резюмируется, и только резюме сохраняется в контексте.

```
[Полная история 10K токенов]
            ↓
      [Резюмирование]
            ↓
    [Компактное резюме 500 токенов]
```

**Преимущества:**
- Сильно экономит токены
- Сохраняет ключевую информацию

**Недостатки:**
- Может потеряться важная деталь
- Требует дополнительного вызова LLM

### Решение 2: Иерархическое хранение контекста

Короткие историии хранятся в памяти агента, длинные — в базе данных, очень длинные — в архиве.

```
[Актуальный контекст (текущая сессия)]
                ↓
        [Краткая история (сегодня)]
                ↓
        [Архив (за неделю)]
                ↓
    [Долговременное хранилище]
```

### Решение 3: Специализированная память

Каждый агент имеет разные типы памяти:
- **Working memory:** текущая задача (несколько минут)
- **Short-term memory:** недавние события (несколько часов)
- **Long-term memory:** важные факты (неопределенный срок)

**Пример:** 
```
Агент аналитик:
- Working: "Сейчас анализирую Q2 финансы"
- Short-term: "Вчера выявил тренд роста"
- Long-term: "Компания работает в SaaS индустрии"
```

### Claude-specific: Работа с контекстным окном 200K

Claude 3.5 Sonnet имеет контекстное окно в 200,000 токенов (примерно 150,000 слов). При проектировании MAS учитывайте:

**Стратегии для использования 200K контекста:**
1. **Агенты-хранилища:** Один выделенный агент может содержать историю всей сессии (200K токенов = ~100 страниц текста)
   ```python
   # Агент для управления историей
   memory_agent = {
       "role": "memory_manager",
       "system_prompt": "You maintain complete session history and provide summaries on request",
       "context_budget": 180000  # Оставить 20K для работы
   }
   ```

2. **Параллельная обработка без переноса контекста:** Специализированные агенты могут работать без доступа к полной истории
   ```python
   specialist_agent = {
       "max_context": 50000,  # Достаточно для узкоспециализированной задачи
       "inputs": ["task", "relevant_data_summary"],  # Не полная история
   }
   ```

3. **Кэширование контекста (Anthropic Prompt Caching):** Используйте кэширование для часто повторяемых частей контекста
   ```python
   # Часто используемая информация кэшируется автоматически
   # Экономит токены и ускоряет обработку
   cached_context = {
       "system_instructions": "...",  # Кэшируется
       "company_info": "...",         # Кэшируется
       "current_session": "..."       # Не кэшируется (изменяется)
   }
   ```

**Рекомендация:** Используйте Claude с 200K контекстом как центральный хранилище состояния, а других агентов как специалистов с ограниченным контекстом (10-50K). Это максимизирует эффективность при минимизации стоимости.

## Часть 4: Обработка ошибок и отказоустойчивость

### Локальная обработка ошибок

Когда агент встречает ошибку, сначала пытается исправить её сам:

1. **Попытка заново:** переформулировать запрос, пересчитать
2. **Резервная стратегия:** использовать альтернативный инструмент
3. **Откат:** отменить последние действия и начать заново

### Эскалация ошибок

Если локально исправить не получилось, ошибка эскалируется:

```
[Агент A] 
   (ошибка, не может исправить)
        ↓
[Координатор]
   (пытается перераспределить задачу)
        ↓
[Агент B] (специалист по этому типу ошибок)
```

### Изоляция отказов

Отказ одного агента не должен разрушить всю систему:

- **Таймауты:** если агент не ответил за N секунд, считается неработающим
- **Fallback-агенты:** если основной агент недоступен, используется резервный
- **Частичные результаты:** система может работать с неполной информацией

**Пример:**
```
Задача: "Проанализировать финансы компании"

Идеально: финансовый агент + агент кода + агент аналитики
Если финансовый агент упал: используется только агент аналитики
Если агент кода не может загрузить данные: используются ручные числа
```

## Часть 5: Оптимизация производительности

### Параллелизм: когда это имеет смысл

Не все задачи можно распараллелить. Параллельное выполнение имеет смысл, когда:

1. **Независимые подзадачи:** результаты не зависят друг от друга
2. **Достаточная сложность:** экономия времени перевешивает overhead координации
3. **Разные инструменты:** агенты не конкурируют за ресурсы

**Пример параллелизации:**
```
Задача: "Подготовить отчет о рынке"

Плохо (последовательно):
[Исследование_1] → [Исследование_2] → [Исследование_3] → [Синтез]
(30 сек + 30 сек + 30 сек + 10 сек = 100 сек)

Хорошо (параллельно):
[Исследование_1]  ┐
[Исследование_2]  ├→ [Синтез]
[Исследование_3]  ┘
(30 сек + 10 сек = 40 сек)
```

### Кэширование результатов

Если один агент уже выполнил определенный анализ, результат кэшируется и переиспользуется другими агентами.

```
[Агент_A] выполняет: "Расчет ROI"
                    ↓
            [Сохранить в кэш]
                    ↓
[Агент_B] нужен ROI → берет из кэша
                    ↓
[Агент_C] нужен ROI → берет из кэша
```

### Адаптивное распределение ресурсов

Система выделяет больше ресурсов (более мощные модели, больше попыток) важным агентам.

```
Критичные операции (SQL запросы): claude-3-5-sonnet-20241022
Вспомогательные операции (резюме): claude-3-haiku-20250305
Кэшируемые операции (факты): локальная модель или cached embeddings
```

## Часть 6: Практический пример: Система аналитики продаж

Давайте рассмотрим конкретный пример мультиагентной системы для анализа продаж:

**Архитектура:**
```
[Пользователь запрашивает анализ]
           ↓
    [Координатор]
     ↙   ↓   ↘
[SQL] [Визуал] [Эксперт]
 Агент Агент   Агент
```

**Рабочий процесс:**

1. **Пользователь:** "Почему продажи упали на 15% в марте?"

2. **Координатор:** 
   - Определяет: нужны данные, визуализация, интерпретация
   - Отправляет SQL-агенту: "Получи данные по продажам за январь-март"
   - Отправляет визуал-агенту: "Когда придут данные, создай график"
   - Отправляет эксперт-агенту: "Проанализируй тренды"

3. **SQL-агент:**
   - Пишет запрос: `SELECT date, sales FROM sales WHERE date >= '2024-01-01'`
   - Возвращает данные координатору
   - Параллельно отправляет в кэш

4. **Визуал-агент:**
   - Получает данные из кэша SQL-агента
   - Создает Matplotlib график
   - Сохраняет как SVG

5. **Эксперт-агент:**
   - Анализирует данные
   - Выявляет: "Падение началось 5 марта, совпало с запуском конкурента"
   - Предлагает рекомендации

6. **Координатор:**
   - Собирает результаты
   - Формирует ответ: "Падение на 15% началось 5 марта. Вероятная причина — запуск конкурента. Рекомендации: [...]"

**Примечание:** В реальной реализации SQL-агент и эксперт-агент могут работать параллельно (асинхронно), пока визуал-агент ждет данных, что сокращает общее время выполнения.

- ✅ Хорошо: "Агент для SQL запросов к БД продаж"
- ❌ Плохо: "Общий агент для всей аналитики"

```
  "constraints": ["не удалять данные", "максимум 30 сек"]

Выходные данные:

  "result": "основной результат",
  "confidence": 0.95,
  "source": "откуда взяты данные",
  "metadata": {"время_выполнения": "2.5s"}
}
```
