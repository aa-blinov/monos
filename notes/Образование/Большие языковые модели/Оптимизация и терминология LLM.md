# Оптимизация и терминология больших языковых моделей

Когда мы загружаем большую языковую модель на GPU, сталкиваемся с двумя главными врагами: **нехватка памяти (VRAM)** и **медленный инференс**. Этот документ объясняет основные техники оптимизации и ключевые термины, которые встречаются при работе с LLM, от теории к практике.

## Часть 1: Память и вычисления — главные узкие места

### Почему память критична

Большая языковая модель — это просто матрица огромных размеров, заполненная миллиардами чисел. Для GPT-4 это несколько триллионов параметров. Когда модель работает, GPU должна хранить в памяти не только саму модель, но и:

1. **Входные данные** — токены, которые мы даём на вход
2. **Промежуточные результаты** — значения, генерируемые в процессе вычисления (activation memory)
3. **Кэш** — значения, сохраняемые для ускорения

Память — это физический ресурс. На RTX 4090 это 24 GB, на M2 Max — 36 GB. Если модель требует 30 GB, она просто не влезет, и GPU вернет ошибку Out of Memory.

### Почему инференс медленный

Инференс — это процесс, где модель генерирует ответ токен за токеном. На каждом шаге она должна:

1. Прочитать всю модель из памяти
2. Выполнить матричные умножения
3. Записать результаты обратно

Каждый дополнительный токен означает ещё один проход через всю модель. Для длинного ответа (500+ токенов) это может занять 10-30 секунд даже на быстром GPU.

## Часть 2: Методы оптимизации памяти

### Flash Attention — революция в обработке внимания

**Что это:** Flash Attention — это переработанный алгоритм вычисления механизма внимания в трансформерах. Обычно внимание работает так: сначала загружается вся матрица внимания (квадратная матрица размером context_length × context_length), затем над ней выполняются операции.

На практике, если контекст 4000 токенов, матрица будет 4000 × 4000 = 16 миллионов элементов. Это огромное количество памяти, которое нужно загрузить, обработать и сохранить.

**Как это работает:** Flash Attention вычисляет эту матрицу **блоками**. Вместо одного большого куска память используется малыми партиями (блоки 256-512 токенов), которые умещаются в **быструю кэш-память GPU** (L2 cache), а не в глобальную VRAM. Это уменьшает количество циклов обращения к памяти в десятки раз.

**На практике:**
- **Скорость инференса:** +30-50% быстрее
- **Потребление VRAM:** -20-30% экономии
- **Качество:** **ноль потери качества** — алгоритм математически эквивалентен стандартному вниманию
- **Требования:** Нужна современная видеокарта NVIDIA (A100, H100, RTX 4090) с поддержкой CUDA

Flash Attention — это одна из первых оптимизаций, которую нужно включить, если вы работаете с длинным контекстом.

### KV-кэш — как модель помнит контекст

**Что это:** Когда трансформер обрабатывает последовательность токенов, для каждого из них вычисляются три матрицы: Query (Q), Key (K) и Value (V). На каждом шаге инференса новый токен должен вычислить свой Q, но для вычисления внимания ему нужны K и V **всех предыдущих токенов**.

KV-кэш — это сохранённые K и V векторы всех предыдущих токенов. Вместо того, чтобы пересчитывать их заново, модель просто берёт их из кэша.

**Пример:** Пусть мы генерируем ответ из 100 токенов. Без KV-кэша:
- Шаг 1 (1 токен): модель обрабатывает 1 токен
- Шаг 2 (2 токена): модель пересчитывает K, V для первого токена И вычисляет для второго
- Шаг 100 (100 токенов): модель пересчитывает K, V для 99 предыдущих И вычисляет для нового

Это **99% избыточной работы**!

С KV-кэшем:
- Шаг 1: вычисляем K, V для 1 токена, сохраняем в кэш
- Шаг 2: берём K, V первого из кэша, вычисляем только для второго
- Шаг 100: берём K, V предыдущих 99 из кэша, вычисляем только для нового

Это **99% экономии вычислений**!

**На практике:**
- **Скорость:** инференс ускоряется в 10-50 раз (в зависимости от длины генерируемого текста)
- **Память:** используется больше (кэш занимает место), но экономия от ускорения перевешивает

KV-кэш — это основной механизм, который делает интерактивные чат-боты возможными. Без него каждое слово в ответе генерировалось бы минуты.

### Квантизация KV-кэша — сжатие памяти

**Что это:** KV-кэш, несмотря на всю его полезность, может потреблять **огромное количество памяти** при работе с длинным контекстом. Для модели в 70B параметров при контексте 32K токенов KV-кэш может занять 40+ GB!

Квантизация — это процесс **уменьшения точности чисел**. Обычно K и V хранятся в формате float16 (16 бит на число). Квантизация переводит их в int8 (8 бит) или даже int4 (4 бита), сжимая данные в 2-4 раза.

**Как это работает:** Вместо хранения точного значения (например, 3.14159), мы храним меньше значащих цифр (например, 3.14), теряя точность, но выигрывая в памяти.

**На практике:**
- **VRAM экономия:** -50-75% потребления памяти для кэша
- **Качество:** почти незаметное снижение (обычно -0.1-0.3% accuracy на бенчмарках)
- **Скорость:** немного медленнее, чем float16, из-за операций конвертации

Квантизация KV-кэша — это **компромисс**: мы теряем небольшую часть качества, но зато можем обрабатывать контексты в 2-4 раза длиннее.

## Часть 3: Вычисления и батчинг

### Evaluation Batch Size — управление параллелизмом

**Что это:** Evaluation Batch Size (часто называется `n_batch` в llama.cpp) определяет, **сколько токенов GPU обрабатывает одновременно** в одном вычислительном шаге.

Modern GPU содержит тысячи ядер. Когда мы обрабатываем один токен, используется только небольшой процент этих ядер. Если же мы обработаем 32 или 64 токена одновременно, ядра будут загружены полностью, и GPU выдаст максимальную производительность.

**Пример:** Представьте ресторан с 100 поварами:
- Если готовить один блюдо одновременно → 99 поваров сидят без дела
- Если готовить 100 блюд одновременно → все повара заняты

**На практике:**
- **Параметры:** обычно 512-2048 для локального инференса, 8192-32768 для серверов
- **Скорость:** увеличение batch size на 2x даёт примерно +30-50% скорости (не линейный прирост из-за других узких мест)
- **Память:** линейное увеличение — batch size 32 занимает в 32 раза больше памяти, чем batch size 1

**Правило:** Установите batch size настолько большим, насколько позволяет ваша VRAM. Это самый простой способ ускорить инференс на 30-50%.

## Часть 4: Архитектурные инновации

### Mixture of Experts (MoE) — модели с триллионами параметров

**Что это:** Mixture of Experts (MoE) — это архитектурный подход, где вместо одного большого слоя нейронной сети используется **набор специализированных подсетей** (экспертов). Специальная функция маршрутизации (Gating Function) решает, какие эксперты должны обработать данный токен.

**Пример из реальной жизни:** Вместо одного универсального врача, который знает обо всём, у вас есть команда специалистов:
- Кардиолог обрабатывает вопросы о сердце
- Невролог обрабатывает вопросы о мозге
- Дерматолог обрабатывает вопросы о коже

Для каждого пациента функция маршрутизации решает: "Нужен кардиолог и дерматолог, но не невролог".

**Как работает MoE в LLM:**

Модель может иметь 1000 экспертов (отдельных подсетей), но для каждого токена активируется только несколько (скажем, 8). Это означает:

1. **Огромное количество параметров:** можно обучить модель с 300B параметров
2. **Низкие вычислительные затраты:** работает как модель с 40B активными параметрами
3. **Специализированное знание:** разные эксперты учатся разным сторонам языка

**Примеры из реальности:**
- Google Switch Transformer: 1.6 триллиона параметров, но только 67B активны на шаг
- Mixtral 8x7B: 47B параметров всего, 13B активны на шаг
- Mistral Large: использует MoE для эффективности

**На практике:**
- **Плюсы:** Can create extremely large models with reasonable computational cost during inference
- **Минусы:** 
  - Сложнее обучать (нужно балансировать нагрузку между экспертами)
  - Может быть неэффективна на задачах, где сложно определить, какой эксперт нужен
  - Требует больше памяти при обучении

**Проблема Gating:** Если маршрутизация работает плохо, все токены могут направиться в одного или двух экспертов, и остальные экспертами будут не использованы. Это называется "коллапс маршрутизации" и сильно снижает эффективность.

## Часть 5: Вместе всё это работает

На практике при запуске LLM локально вы можете увидеть:

```
llama.cpp with:
- Flash Attention enabled ✓
- KV cache quantization: Q8_0
- n_batch: 512
- Loaded model: Mixtral 8x7B
```

Что это означает:

1. **Flash Attention** — каждый проход внимания происходит очень быстро
2. **KV cache quantization Q8_0** — контекст хранится в 8-битном формате, экономя 50% памяти
3. **n_batch: 512** — GPU обрабатывает 512 токенов параллельно для максимальной производительности
4. **Mixtral 8x7B** — модель использует MoE, поэтому запускается быстро, несмотря на внушительный размер

Вместе эти техники позволяют запустить 70B+ модель на потребительском GPU (RTX 4090) и получить приемлемую скорость генерации (30-50 токенов в секунду).

## Часть 6: Практические рекомендации

### Если у вас мало VRAM (< 16 GB)

1. Включите квантизацию модели (Q5_K_M, Q4_K_M)
2. Снизьте n_batch до 128-256
3. Используйте Flash Attention (если поддерживается)
4. Работайте с контекстом не больше 2048 токенов
5. Включите KV cache quantization

### Если у вас среднее VRAM (16-32 GB)

1. Загрузите модель в Q5_K_M или даже в fp16
2. Установите n_batch в 512-1024
3. Используйте Flash Attention
4. Контекст до 8192 токенов
5. KV cache quantization опционально

### Если у вас много VRAM (32+ GB)

1. Загрузите модель в fp16 или даже bfloat16
2. Установите n_batch в 2048-4096
3. Используйте Flash Attention
4. Контекст до 32768+ токенов
5. KV cache quantization не нужна

### Как выбрать размер модели

- **7B**: хорошо для простых задач (классификация, краткие ответы) на GPU 4GB+
- **13B**: универсальный выбор, работает на 8GB
- **70B**: требует 24GB+ VRAM, но значительно лучше качество
- **Mixtral 8x7B**: лучший компромисс качество/скорость/память
- **300B+ MoE**: только на серверах

## Словарь терминов

- **VRAM:** видеопамять GPU, физический ресурс
- **Infer ence:** процесс генерации ответа (отличается от training)
- **Token:** единица текста (примерно 1 слово = 1.3 токена)
- **Batch:** группа токенов, обрабатываемых одновременно
- **Float16/FP16:** число с 16-битной точностью
- **Int8:** целое число 8-битной точности (квантизация)
- **Gating Function:** функция, которая решает, какие эксперты использовать в MoE
- **Activation Memory:** память, занимаемая промежуточными результатами во время вычисления

## Заключение

Оптимизация LLM — это баланс между тремя факторами:

1. **Качество** — точность и логичность ответов
2. **Скорость** — как быстро генерируется ответ
3. **Памяти** — сколько VRAM требуется

Нет идеального решения, но комбинация Flash Attention + KV cache + оптимальный batch size + квантизация позволяет запустить конкурентоспособную модель на почти любом современном GPU.
