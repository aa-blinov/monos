# Retrieval-Augmented Generation: Полный обзор

## Введение: Проблема современных LLM и её решение

Большие языковые модели (LLM) вроде GPT-4 или LLaMA демонстрируют впечатляющие способности в генерации текста и решении задач естественного языка. Однако они сталкиваются с фундаментальными ограничениями:

1. **Галлюцинации** — модель генерирует правдоподобный, но ложный контент
2. **Устаревшие знания** — параметры модели содержат информацию только до момента обучения
3. **Отсутствие источников** — невозможно проследить, откуда модель взяла информацию
4. **Неточность в фактических задачах** — на knowledge-intensive задачах параметрические модели отстают от специализированных архитектур

Retrieval-Augmented Generation (RAG) — это парадигма, которая решает эти проблемы, объединяя параметрическую память LLM с явной непараметрической памятью (обычно векторной базой данных).

## 1. Основные компоненты RAG системы

### 1.1 Трёхкомпонентная архитектура

RAG состоит из трёх взаимосвязанных компонентов:

```
┌─────────────────────────────────────────────────────────┐
│                    RAG СИСТЕМА                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [Retrieval]     [Augmentation]      [Generation]      │
│       ↓                  ↓                   ↓          │
│   Поиск в         Интеграция         Синтез ответа     │
│   базе            контекста          на основе         │
│   данных          с запросом         найденного        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 Компонент Retrieval (Поиск)

**Назначение:** Найти наиболее релевантные документы/фрагменты из внешней базы данных по запросу пользователя.

**Ключевые технологии:**

- **Dense Retrieval** (плотный поиск)
  - Использует neural retriever (например, DPR — Dense Passage Retrieval)
  - Кодирует запрос и документы в плотные векторные представления
  - Находит близайшие векторы по метрике сходства (cosine similarity)
  - Преимущество: схватывает семантическое сходство, а не просто ключевые слова

- **Sparse Retrieval** (разреженный поиск)
  - Традиционные BM25, TF-IDF
  - Быстрее, но менее гибкий
  - Часто комбинируется с dense retrieval для лучших результатов

- **Hybrid Retrieval**
  - Комбинирует dense и sparse методы
  - Обычно даёт лучшие результаты на практике

**Пример работы:**

```
Запрос: "Кто изобрел транзистор?"

↓

Embedding: query_vector = encoder("Кто изобрел транзистор?")

↓

Поиск: top-k документов с максимальной косинус-схожестью

↓

Результат:
[
  "Уильям Шокли, Джон Бардин и Леон Брейтвейн изобрели первый 
   работающий транзистор в 1947 году в Bell Labs...",
  "Хронология развития полупроводниковых приборов...",
  "Влияние транзистора на развитие электроники..."
]
```

### 1.3 Компонент Augmentation (Дополнение)

**Назначение:** Объединить найденные документы с исходным запросом в структурированную подсказку для LLM.

**Основные подходы:**

- **Concatenation** (конкатенация)
  ```
  Контекст: [Найденный документ 1] [Найденный документ 2]
  Запрос: [Исходный вопрос пользователя]
  
  → Передается в LLM как единая подсказка
  ```

- **Structured Augmentation** (структурированное дополнение)
  ```
  System: Ты — помощник, отвечающий на основе предоставленного контекста.
  
  Контекст из базы данных:
  - Факт 1: ...
  - Факт 2: ...
  - Источник: Википедия, раздел "История транзистора"
  
  Вопрос пользователя: Кто изобрел транзистор?
  ```

- **Prompt Engineering для RAG**
  - Явная инструкция: "Используй только предоставленный контекст"
  - Указание формата ответа
  - Просьба о ссылке на источник

### 1.4 Компонент Generation (Генерация)

**Назначение:** На основе запроса и контекста сгенерировать точный, обоснованный ответ.

**Особенности в RAG:**

- LLM теперь работает как **экспертный синтезатор**, а не источник знаний
- Может условно ссылаться на контекст ("Согласно найденному документу...")
- Менее склонна к галлюцинациям (есть факты для проверки)
- Может генерировать разные варианты ответа, выбирая лучший

**Метрики качества:**

- ROUGE (совпадение n-грамм с эталонным ответом)
- BLEU (для machine translation, применима и здесь)
- Factual Correctness (проверка фактов)
- Relevance (релевантность найденному контексту)

## 2. Эволюция RAG парадигм

### 2.1 Naive RAG (базовый подход)

**Схема:**

```
Запрос → [Retriever] → Найденный контекст
                          ↓
                    Prepend to prompt
                          ↓
         Запрос + Контекст → [LLM] → Ответ
```

**Характеристики:**

- Простой и понятный pipeline
- Поиск выполняется один раз перед генерацией
- Контекст используется "как есть" без обработки
- Время отклика ограничено скоростью retrieval + inference LLM

**Проблемы:**

- Качество результата сильно зависит от качества поиска
- Если retriever ошибся, генератор не может исправить
- Контекст может быть слишком большим или слишком маленьким
- Нет взаимодействия между retriever и generation этапами

**Когда работает хорошо:**

- Простые factual вопросы ("Когда произошло событие X?")
- Когда контекст очень релевантен
- На больших и хорошо организованных базах данных

### 2.2 Advanced RAG (продвинутый подход)

Решает проблемы Naive RAG через:

#### а) Улучшение качества retrieval

**Query Expansion** (расширение запроса)
```
Исходный запрос: "LLM"
         ↓
Расширенные варианты:
- "Large Language Models"
- "Большие языковые модели"
- "neural language models"
- "foundation models"
         ↓
Поиск по всем вариантам и объединение результатов
```

**Multi-hop retrieval** (многошаговый поиск)
```
Вопрос: "Какой компании принадлежит технология трансформера?"
         ↓
Шаг 1: Найти "трансформер" → Google Brain
         ↓
Шаг 2: Найти "Google Brain" → Google/Alphabet
         ↓
Ответ: Google (через Alphabet)
```

**Chunk Optimization** (оптимизация размера контекста)
```
Вместо целого документа → использовать оптимальный размер фрагмента

Слишком маленький chunk: потеря контекста
Слишком большой chunk: шум и диффузия релевантной информации

Рекомендация: 100-300 слов с перекрытием 20-30%
```

#### б) Итеративное улучшение

**Query Rewriting** (переформулировка запроса)
```
Пользователь: "это про тот новый способ"
LLM переформулирует: "Какие недавние методы повышают точность LLM?"
↓ (более точный поиск)
Лучшие результаты
```

**Re-ranking** (переранжирование)
```
Retriever находит top-k = 100 документов

Cross-encoder переранжирует на основе:
- Тематической релевантности
- Доверительности источника
- Новизны информации

Финальный top-k = 5-10 документов для генерации
```

#### в) Фильтрация контекста

**Активная фильтрация:**
```
Найденные документы → Фильтр:
  ✓ Содержит прямой ответ?
  ✓ Достаточно свежий? (дата публикации)
  ✓ Из надежного источника?
  ✓ Не противоречит другим источникам?
          ↓
    Отфильтрованный контекст → LLM
```

### 2.3 Modular RAG (модульный подход)

Наиболее гибкий подход, позволяющий:

```
┌─────────────────────────────────────────────────────────┐
│           MODULAR RAG ARCHITECTURE                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [Query Processing] → [Routing Module]                 │
│                            ├→ [Knowledge Retrieval]    │
│                            ├→ [Tool Calling]           │
│                            └→ [Web Search]             │
│                                    ↓                    │
│                          [Synthesis Module]            │
│                                    ↓                    │
│                      [Output Generation]               │
│                                    ↓                    │
│                          [Evaluation & Feedback]       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Ключевые особенности:**

- **Маршрутизация**: LLM решает, какой источник использовать
  - Локальная база данных компании?
  - Интернет поиск?
  - API специализированного сервиса?
  - Все вместе?

- **Специализированные модули**: не один retriever для всего
  - Для технической документации — одна база
  - Для новостей — другая (с свежей информацией)
  - Для кода — GitHub Search через API

- **Адаптивный контекст**: размер и тип контекста зависит от задачи
  - Для глубокого анализа: большой контекст
  - Для быстрого ответа: минимальный контекст
  - Для творческих задач: минимум контекста (больше свободы модели)

**Пример work flow:**

```
Вопрос: "Какие последние достижения в области квантовых компьютеров?"

↓

Router: "Это требует свежих новостей"

↓

Actions:
1. WebSearch("quantum computing breakthroughs 2024")
2. DBSearch(academic_papers, "quantum computing")
3. Tool: arXiv API for latest papers

↓

Synthesizer: объединяет результаты, убирает дубликаты

↓

LLM: генерирует развёрнутый ответ с ссылками на источники
```

## 3. Реальные примеры использования RAG

### 3.1 Question Answering (Open Domain QA)

**Задача:** Ответить на вопрос на основе большой коллекции текстов (например, Wikipedia).

**Архитектура:**
```
Вопрос: "Какой город является столицей Франции?"
    ↓
Dense Retriever (DPR) + BM25:
    ├─ Найти статьи про Францию
    ├─ Найти статьи про столицы европейских стран
    └─ Объединить и переранжировать
    ↓
Контекст: "Париж — столица Франции, расположен в северо-центральной части..."
    ↓
Generator (T5, BART, LLaMA):
    ├─ Input: [Контекст] + [Вопрос]
    └─ Output: "Париж"
```

**Результаты (из оригинальной статьи Lewis et al., 2020):**
- Natural Questions dataset: **68.4%** (RAG) vs 51.4% (baseline)
- TriviaQA: **56.5%** (RAG) vs 47.8% (baseline)
- WebQuestions: **41.5%** (RAG) vs 34.7% (baseline)

### 3.2 Fact Checking и Claim Verification

**Задача:** Проверить, верно ли утверждение на основе известных фактов.

**Pipeline:**

```
Утверждение: "Никола Тесла изобрел переменный ток"
    ↓
Query formulation: "Кто изобрел переменный ток? История AC"
    ↓
Retrieval: статьи про историю электричества
    ↓
Evidence: "Переменный ток был открыт Майклом Фарадеем в 1831..."
    ↓
Verification: 
    - Supporting: "Тесла совершенствовал и популяризировал AC"
    - Refuting: "AC не был изобретен Теслой"
    - Neutral: "Дискуссия о приоритетах в науке"
```

### 3.3 Summarization (Суммаризация)

**Задача:** Создать краткое резюме на основе больших документов.

**RAG-enhanced summarization:**

```
Документ (3000 слов):
    ↓
Поиск ключевых фрагментов (retriever):
    - Введение (зачем документ)
    - Основные результаты
    - Выводы
    ↓
Контекст: [введение] + [ключ. результаты] + [выводы]
    ↓
LLM: синтезирует когерентное резюме (200-300 слов)
```

**Преимущество над обычной LLM:**
- Не теряет важную информацию
- Более объективно (опирается на факты)
- Быстрее (обрабатывает только ключевые части)

### 3.4 Аналитические задачи (Analytics & Insights)

**Задача:** Дать аналитический отчет на основе набора документов.

**Пример: Анализ отзывов клиентов**

```
Входные данные: 10,000 отзывов о продукте

RAG Pipeline:
1. Retrieval: группировка отзывов по темам
   - Качество продукта: 3500 отзывов
   - Доставка и логистика: 2800 отзывов
   - Поддержка клиентов: 1200 отзывов
   - Цена: 1500 отзывов

2. Augmentation: выбор репрезентативных примеров
   - Позитивные примеры из каждой группы
   - Негативные примеры (для понимания проблем)
   - Нейтральные (для контекста)

3. Generation: LLM создает отчет
   - Основные выводы
   - Рекомендации по улучшению
   - Приоритеты действий
```

## 4. Технические детали и best practices

### 4.1 Выбор модели retriever

**Dense Retrieval (DPR, Contriever):**
- ✓ Схватывает семантическое сходство
- ✓ Современный и точный
- ✗ Требует больше вычислений
- ✗ Может быть медленнее на первый запрос

**Sparse Retrieval (BM25):**
- ✓ Быстро
- ✓ Хорошо работает с техническими терминами
- ✗ Менее точен на естественном языке
- ✗ Не схватывает синонимы и парафразы

**Hybrid (BM25 + Dense):**
- Обычно лучший выбор
- Можно взвешивать результаты

### 4.2 Оптимизация размера контекста

**Проблема: Context Window**

```
Человеческий контекст: ~10,000 слов (комфортно читать)
LLM контекст (GPT-3.5): 4,096 токенов (~3,000 слов)
LLM контекст (GPT-4): 8,192 или 128,000 токенов

Вывод: нужно выбирать контекст тщательно!
```

**Стратегии:**

1. **Per-token retrieval** (из оригинальной RAG)
   - Разные документы могут использоваться для разных позиций в выходе
   - Более точно, но медленнее

2. **Hierarchical retrieval**
   ```
   Шаг 1: Найти релевантный раздел (большие чанки)
   Шаг 2: В найденном разделе выбрать точный фрагмент (малые чанки)
   ```

3. **Adaptive chunking**
   ```
   Документ → [Анализ структуры] → Оптимальные границы чанков
   
   Не просто: каждые 300 слов
   А: учитывать параграфы, заголовки, логические переходы
   ```

### 4.3 Обработка многоязычных запросов

**Challenge:** Документы на одном языке, вопрос на другом

**Решения:**

```
Вопрос на русском: "Какой город столица Франции?"

Вариант 1: Переводный retrieval
  ↓ translate → "What city is the capital of France?"
  ↓ retrieve → English documents
  ↓ answer

Вариант 2: Multilingual embeddings (BERT-multilingual, LaBSE)
  ↓ Embedding model понимает оба языка
  ↓ Напрямую ищет в документах на любом языке

Вариант 3: Cross-lingual retrieval
  ↓ Запрос на русском → поиск в документах на английском, французском, немецком
  ↓ Синтез ответа с учетом языка пользователя
```

## 5. Вызовы и текущие исследования

### 5.1 Проблемы

**1. Когда retrieval ошибается**

```
Вопрос: "Какой континент находится ниже Европы?"

Наивный retriever ищет "ниже Европы" → технические документы об IP адресах
Результат: Бред

Решение: Context-aware retrieval, семантическое понимание
```

**2. Poisoning и hallucinations в контексте**

```
Если база данных содержит ошибочную информацию:
  ✗ RAG будет генерировать на основе этой ошибки
  ✗ Нельзя заранее знать все ошибки в базе

Решение: Fact verification module, multiple source cross-checking
```

**3. Стоимость inference**

```
Традиционный LLM: 1 inference pass
RAG system: 
  - 1 retrieval query
  - 1-2 LLM passes для генерации
  = 2-3x дороже по вычислениям
```

**4. Outdated knowledge problem**

```
Документ в базе: "COVID-19 началась в 2020 году, смертность 2%"
Сегодня: это уже не актуально

Нужна: автоматическая переиндексация, версионирование контекста
```

### 5.2 Открытые исследовательские направления

1. **More efficient retrieval**
   - Субсекундный поиск в миллиардах документов
   - Локальные модели для retrieval

2. **Better fusion of multiple sources**
   - Как объединять противоречивые источники?
   - Как оценивать надежность источника?

3. **Long-context generation**
   - Генерация длинных ответов, согласованных с контекстом

4. **Grounding in external knowledge**
   - Интеграция с графами знаний, структурированными БД

5. **Multimodal RAG**
   - Поиск и генерация с изображениями, видео, аудио

## 6. Практическое внедрение RAG

### 6.1 Стек технологий

**Минимальный стек:**
```
Vector DB: FAISS, Milvus, Pinecone
LLM: Open-source (LLaMA) или API (OpenAI)
Framework: LangChain, Llamaindex
```

**Production-grade стек:**
```
Vector DB: Elasticsearch + ANCE embeddings
LLM: Fine-tuned open-source model
Orchestration: Airflow (для переиндексации)
Monitoring: Prometheus + Grafana
```

### 6.2 Метрики оценки

```
1. Retrieval metrics:
   - MRR (Mean Reciprocal Rank)
   - NDCG (Normalized Discounted Cumulative Gain)
   - Recall@k

2. Generation metrics:
   - ROUGE (for summarization)
   - BLEU (for translation-like tasks)
   - Human evaluation (for quality)

3. End-to-end metrics:
   - Factual correctness
   - User satisfaction
   - Latency
```

## 7. Заключение

RAG представляет собой фундаментальный сдвиг в подходе к knowledge-intensive NLP задачам. Вместо того чтобы требовать от LLM "знать всё", мы даём ей инструмент поиска и просим синтезировать ответ на основе найденной информации.

**Ключевые выводы:**

1. **RAG решает реальные проблемы** LLM: галлюцинации, устаревшие знания, отсутствие источников

2. **Архитектура развивается:** от простого Naive RAG через Advanced RAG к гибкому Modular RAG

3. **Это не просто "поиск + LLM"** — требует тщательной оптимизации каждого компонента

4. **Будущее за гибридными системами**: LLM + structured knowledge + tools

5. **Практическое применение уже здесь**: QA, fact-checking, summarization, analytics

Для практиков: начните с простого pipeline (BM25 + LLM), затем добавляйте complexity по мере необходимости. Не каждой системе нужен модульный RAG, но каждой может помочь базовый retrieval.
